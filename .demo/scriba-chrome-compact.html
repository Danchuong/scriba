Reclaiming the space the controls bar and narration box waste

The widget spends a full padded top bar (prev/next + counter + dots) and a bordered bottom box (min-height 2.5rem) on chrome. Four ways to shrink it — same scene, same visual system, shown against the current baseline.

Today's chrome in scriba-embed.css: .scriba-controls = padding:.6rem 1rem + bg + bottom border (~44 px); .scriba-step-counter = min-width:5rem (far wider than "5 / 5" needs); .scriba-narration = padding:.75rem 1rem + top border + min-height:2.5rem (~64 px). Combined that's ~108 px of chrome wrapping a small animation.

Suggested: ship C3 now — pure CSS, ~54% chrome reduction, zero markup/JS change, no risk to the step controller. If you want it tighter and are OK touching the emit markup, C2 (unified footer) or C4 (edge progress) cut chrome by ~60–70%. C1 overlay looks slickest but the pill over a small stage can crowd content.
