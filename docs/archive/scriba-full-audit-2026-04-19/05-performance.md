# Performance Audit — Scriba

**Date**: 2026-04-19
**Method**: Static analysis only
**Scope**: Render pipeline hot paths, primitive frame generation, large-input scaling
**Baseline**: Post wave 8 round C (`965d705`)

---

## 1. Score

**6.5 / 10**

The wave 8 round C work eliminated two clear wins (npm-root subprocess, KaTeX macros copy, Graph addressable-parts O(n) traversal per call). What remains are structural issues that were deliberately deferred: per-call regex compilation inside hot loops, `inspect.signature` introspection inside the per-frame SVG pass, `copy.deepcopy` on every frame with bindings, and a viewbox/annotation pre-pass that scales O(F × P) rather than O(F + P). These are not hypothetical; they land on every render of an animation with more than a handful of frames.

## 2. Hot Path Map — Top 5 Functions Likely to Dominate Runtime

| 1 | `_emit_frame_svg` | `scriba/animation/_frame_renderer.py:352` | Called once per frame per render pass. | 2 | `emit_interactive_html` / `emit_animation_html` | `scriba/animation/_html_stitcher.py:303` / `scriba/animation/_html_stitcher.py:79` | Drives the viewbox pre-scan loop: `compute_viewbox` is called once per frame plus once for the base, then a second nested O(F × P) pass for `_arrow_height_above`. | 4 | `_render_source` / `_render_cell` | `scriba/tex/renderer.py:510` / `scriba/tex/renderer.py:376` | The TeX pipeline executes 10+ sequential `re.sub()` / `str.replace()` passes over the full source string. `apply_typography` compiles three patterns per call. `apply_size_commands` compiles its combined switch regex (`switch_re`) fresh every call. `_replace_balanced` inside `apply_text_commands` compiles a new pattern per command per call (9 commands × calls). |
| 5 | `_preprocess_text_command_chars` | `scriba/tex/parser/math.py:32` | Called once per math expression extracted from the source. ## 4. Memory-Pressure Findings

| Severity | Location | Pattern | Estimated Pressure | Fix |
|----------|----------|---------|--------------------|-----|
| HIGH | `scriba/animation/_html_stitcher.py:402-527` (`emit_interactive_html`) | The function accumulates four parallel lists in memory simultaneously: `_frame_parts` (list of 4-tuples of escaped SVG strings), `_json_frames_raw` (list of dicts with raw SVG strings), `print_frame_items` (list of strings with SVG + narration), and `_manifests` (list of JSON strings). For a 100-frame animation, each SVG string may be 5-50 KB. At 100 frames × ~20 KB average × 4 copies = ~8 MB of live strings before any list is released. | 5-10 MB peak RSS for a large animation; strings are not released until the function returns. | `_json_frames_raw` is only needed when `inline_runtime=False`. The SVG string in `_frame_parts` and `_json_frames_raw` are the same value; store a single reference and avoid duplicating the string. |
| HIGH | `scriba/animation/_frame_renderer.py:56-109` (`_prescan_value_widths`) | Performs `copy.copy(prim._values)` and `list(prim.cells)` for every primitive before scanning all frames. | MEDIUM | `scriba/animation/scene.py:239-260` (`SceneState.snapshot`) | At each frame, allocates a fresh dict-of-dicts copy of `shape_states` and a fresh `frozenset` of highlights. Over 100 frames, this creates 100 separate dict objects in memory, each holding 100 `ShapeTargetState` instances. Python's garbage collector can handle this, but GC pressure from short-lived dict allocations is real. | For 100 frames × 100 targets × 3 primitives: 30,000 `ShapeTargetState` instances allocated and then dereferenced. | MEDIUM | `scriba/animation/_html_stitcher.py:262-268` (`emit_substory_html`) | `json_frames` is a list of dicts, each containing the full SVG string for a substory frame. ## 5. Top 3 Priorities

### Priority 1 — Hoist all per-call `re.compile` calls to module-level constants

**Files affected**: `scriba/tex/parser/math.py`, `scriba/tex/parser/dashes_quotes.py`, `scriba/tex/parser/text_commands.py`, `scriba/core/text_utils.py`, `scriba/animation/_frame_renderer.py`

**Rationale**: This is the highest effort-to-reward fix in the codebase. At least 7 distinct patterns are compiled inside function bodies that are called on every render. The TeX pipeline calls `apply_typography`, `apply_size_commands`, and `apply_text_commands` for every block-level paragraph and every table cell. The animation pipeline compiles 3 regexes per primitive per frame in `_expand_selectors`. None of these patterns are data-dependent; they are all invariant at module import time. Python's internal regex cache provides a partial mitigation, but cache lookup itself has overhead (hash + lock) that module-level constants eliminate entirely. Fixing all 7 sites is a single-pass mechanical refactor with no behavior change and no test impact beyond performance. Conservative estimate: 5-15% total render time reduction on TeX-heavy documents with tables.

### Priority 2 — Replace `inspect.signature` in `_emit_frame_svg` with a class-level boolean flag

**File**: `scriba/animation/_frame_renderer.py:384`

**Rationale**: `inspect.signature` is called inside the innermost loop of the animation render pass — once per primitive per frame. This is the function that dominates frame generation time. The question being asked (`does apply_command accept target_suffix?`) is a property of the class, not the instance, and does not change at runtime. Moving it to a class-level attribute or a module-level `dict[type, bool]` cache eliminates 90+ `inspect.signature` calls per 30-frame render. This is a two-line fix in `PrimitiveBase` (add a `_apply_command_accepts_suffix: ClassVar[bool] = False` default) and a one-line override in each primitive that does accept the suffix. The frame renderer then reads `type(prim)._apply_command_accepts_suffix` instead of calling `inspect.signature`.

### Priority 3 — Flatten the dual O(F × P) viewbox pre-scan into a single O(F + P) pass

**File**: `scriba/animation/_html_stitcher.py:113-148` and `scriba/animation/_html_stitcher.py:349-381`

**Rationale**: Before any SVG is emitted, both `emit_animation_html` and `emit_interactive_html` run two nested loops over frames and primitives: one to compute the max viewbox, and one to compute `set_min_arrow_above`. These can be merged into a single pass. More importantly, the inner `compute_viewbox` call invokes `prim.bounding_box()` and `prim.set_annotations()` on every primitive for every frame. For `Graph` with force-directed layout, `bounding_box()` triggers `arrow_height_above()` which iterates annotations. Merging the two loops and short-circuiting primitives with no annotations (detectable in O(1) from the frame data) reduces the pre-scan from O(F × P × A) to O(F × P) at worst, and to O(F + P) for annotation-free animations. This also removes the code duplication between the two emit functions, which currently have identical pre-scan logic copy-pasted verbatim.

## Appendix: What Wave 8 Round C Already Addressed

For context, the following were fixed in `965d705` and should not be re-reported:

- `npm root -g` subprocess on every `TexRenderer.__init__` — fixed (skipped when vendored KaTeX present)
- `_discover_node_global_root` called multiple times per process — fixed (`lru_cache(maxsize=1)`)
- `self._katex_macros` dict copied on every `_render_inline` / `_render_display` call — fixed (`_katex_macros_request_dict` pre-allocated in `__init__`)
- `Graph.addressable_parts` O(n) traversal called on every selector validation — fixed (instance-dict cache invalidated on topology mutation)
- KaTeX CSS inlined even for math-free documents — fixed (presence scan gates `inline_katex_css`)
